# Modernization And AI Enhancement Plan

## Scope

- Snapshot date: 2026-06-17.
- Goal: modernize and refactor IsleMind while preserving current behavior, then use `docs/architecture/ai-technology-landscape.md` as a gated enhancement radar.
- This is a planning and parity document. It is not approval to change public APIs, upgrade frameworks, or add heavy runtime dependencies.
- Current worktree is dirty. Start implementation by separating existing user changes from each refactor pass.

## AI Access Layer Goal

Upgrade IsleMind's AI access layer into a broad provider/model compatibility platform that can:

- cover a large provider and model catalog instead of only a few common defaults,
- support official, relay, aggregator, and local-runtime chains without silently assuming one universal protocol,
- preserve response, cache, compaction, tool, multimodal, retrieval, and reasoning behavior through capability-aware routing,
- explain unsupported or downgraded paths through UI, traces, tests, and docs.

The target state is not "many providers can be added". The target state is "provider capabilities are explicit, testable, and understandable".

## AI Access Capability Matrix

The active access-layer scope is broader than response calls, cache, remote compact, transport, and fallback. The compatibility program should explicitly cover these dimensions:

| Dimension | Required outcome |
| --- | --- |
| Provider coverage | Catalog official vendors, cloud-hosted vendors, aggregators, relays, and local OpenAI-compatible runtimes. |
| Model coverage | Track model ids, aliases, deprecations, preferred endpoints, tool support, reasoning mode, and verification date. |
| Authentication | Support provider-specific key/header/base-URL/region/tenant semantics without assuming one credential shape. |
| Protocol semantics | Distinguish Responses, chat-completions, Anthropic Messages, Gemini generateContent, and provider-specific compatible variants. |
| Transport semantics | Correctly negotiate HTTP, SSE, and WebSocket paths, plus transport fallback reasons. |
| Multi-turn continuity | Preserve response ids, replay items, tool-call state, and conversation continuation across supported providers. |
| Context management | Support local compression, remote compact, threshold policy, truncation, and state reuse with explicit decision logs. |
| Cache behavior | Support provider-native cache semantics where available and fail closed with clear explanation where unavailable. |
| Tool chain | Cover provider-native tools, function calling, MCP, search, and replay/reconciliation behavior. |
| Multimodal I/O | Handle image, file, audio, speech, and model-specific modality blocking or shaping. |
| Retrieval stack | Preserve embeddings, rerank, RAG injection, and citation behavior across provider families. |
| Observability | Surface declared, ready, observed, downgraded, and blocked capability states in diagnostics and traces. |
| Import/probe lifecycle | Detect presets, infer safe defaults, suppress unsupported sync paths, and keep model catalogs current. |
| Complex chain compatibility | Validate official -> relay -> aggregator -> IsleMind and local runtime -> gateway -> IsleMind style chains. |

## Refactor Principles

- Preserve behavior unless a pass explicitly declares a functional change and gets separate approval.
- Keep public entry points stable first: `streamChat`, `generateText`, `testProviderModelDetailed`, `fetchProviderModelConfigsDetailed`, `syncProviderCredentialGroupsDetailed`, `sendMessage`, `retryMessage`, `regenerateLastAssistant`, and `stopMessage`.
- Prefer extraction, adapter seams, and pure helper tests before moving runtime orchestration.
- Keep each pass reviewable. A pass should touch one boundary and one validation set.
- Do not treat external AI projects as missing capabilities until the current source path has been inspected.

## AI Technology Landscape Feasibility

The AI technology landscape is feasible as a maintained adoption radar because it already matches IsleMind's current architecture:

- Local-first constraint: current retrieval and local model paths support offline and resource-aware behavior.
- Controlled execution: current MCP and agent paths already expose permission and trace boundaries.
- Explainability: existing process traces, RAG evaluation logs, and QA evidence scripts can host acceptance evidence.
- Cancellability: chat runtime and tool execution should remain bounded before any broader agent autonomy.
- Platform fit: React Native runtime remains the default boundary; heavy parsing, serving, eval, and observability tools should start as local dev tools or optional service integrations.

The main feasibility risk is not capability absence. It is integration blast radius: several relevant systems are already advanced but concentrated in oversized modules. Enhancement should therefore add evaluation harnesses and diagnostics before replacing runtime paths.

## Hosted Provider Boundary

The current runtime now distinguishes between:

- official providers,
- aggregators,
- relays,
- local runtimes,
- cloud-hosted compatibility chains.

Cloud-hosted compatibility chains are not treated as generic "OpenAI-compatible" success cases. In particular:

- Azure OpenAI v1 has partial support through `/openai/v1`, while deployment-name legacy URLs still need hosted-path aware handling.
- AWS Bedrock Mantle OpenAI-compatible `/v1` paths are partial-ready for chat, Responses, and model-list routing with Bedrock API-key bearer auth.
- AWS Bedrock Runtime can prepare signed non-streaming `InvokeModel` requests for Anthropic-style bodies when AWS credentials are supplied; full direct Runtime support still needs model-list sync, streaming, Converse, remote compact, and provider-specific cache/tool behavior.
- Vertex AI OpenAI-compatible `/endpoints/openapi` paths are partial-ready; native Vertex Gemini paths still need hosted auth and path-aware routing.

These providers should surface as explicit planned or partial support until their runtime path, model discovery, cache behavior, and diagnostics are implemented end to end.
